# FN-10.1.1.3: SSO Konfigurieren

> **Parent**: [CMP-10.1.1](../components/CMP-10.1.1_Mandanten_Verwaltungsmodul.md)

---

## Functional Description

Das System muss pro Mandant SAML 2.0 oder OpenID Connect konfigurierbar machen: Entity-ID, Metadata-URL, Certificate. Test-Login vor Aktivierung.

- Das System soll beide Protokolle (SAML 2.0, OIDC) unterstuetzen.
- Das System soll eine Test-Login-Funktion bieten (ohne Auswirkung auf Produktivdaten).
- Das System soll das Certificate als Upload oder URL akzeptieren.

---

## Preconditions

- Mandant existiert.
- Admin hat SSO-Daten des Identity-Providers.

---

## Behavior

1. Admin waehlt das Protokoll: SAML 2.0 oder OIDC.
2. Bei SAML 2.0: Admin gibt Entity-ID, Metadata-URL und Certificate (Upload/URL) ein.
3. Bei OIDC: Admin gibt Client-ID, Client-Secret, Issuer-URL und Redirect-URI ein.
4. System validiert die Eingaben (Metadata-URL erreichbar, Certificate parsbar).
5. Admin klickt "Test-Login": System fuehrt einen Test-Authentifizierungsflow durch.
6. Bei erfolgreichem Test: Admin kann SSO aktivieren.
7. System protokolliert die SSO-Konfiguration als Audit-Event.

---

## Postconditions

- SSO ist konfiguriert und optional aktiviert.
- Test-Login war erfolgreich (bei Aktivierung).
- Audit-Event ist protokolliert.

---

## Error Handling

- Das System soll bei nicht erreichbarer Metadata-URL die Meldung "Metadata-URL nicht erreichbar" anzeigen.
- Das System soll bei fehlgeschlagenem Test-Login die Fehlermeldung des IdP anzeigen.
- Das System soll bei ungueltigem Certificate die Meldung "Certificate konnte nicht geparst werden" anzeigen.

---

## Acceptance Criteria (functional)

- [ ] SAML 2.0 Konfiguration mit gueltiger Metadata-URL wird akzeptiert.
- [ ] OIDC Konfiguration mit gueltigem Issuer wird akzeptiert.
- [ ] Test-Login fuehrt einen echten Auth-Flow durch.
- [ ] Nicht erreichbare Metadata-URL wird mit Fehlermeldung abgelehnt.
- [ ] SSO kann nach erfolgreichem Test aktiviert werden.
